DOORS: Paradox Building Rules

Contents

    Building Standards for DOORS: Paradox

    Everything here is a requirement. If a build doesn't meet these rules, the room will NOT be added to the game. Keep scale, distance, assets, and hierarchy consistent across all rooms.

    Under no circumstance should you ignore these rules, if these rules are too difficult to understand for you, you might not be suitable to be a builder for Paradox, this is not being rude, this is enforcing quality.

    Basics

    #

    Grid & Alignment
    • All transforms must be multiples of 0.125 studs. (…0.125, 0.25, 0.5, 1.0, etc.)
    • Never use free-scale/floating values like 0.373 unless explicitly called out in the spec for a prop.
    • Pivots anchored to logical center or bottom of assets. Doors: hinge pivot aligned to hinge edge.
    • No face overlaps or “z-fighting”. No micro-nudges “to look right”. Make the geometry right.

    When “off-grid” is allowed
    • Decor only: subtle rotation or offset of furniture to break repetition.
    • Never for structure (walls, floors, ceilings, trims, pillars, door/window frames).

    Room Basics

    #

    Start & Exit orientation
    • Start part faces outside of the main room.
    • Exit part faces inside of the main room.
    • Both must be on-grid and flush with door thresholds; no micro gaps.

    RoomEntity_SpawnNode
    • Required in every room.
    • Place in front of Start or roughly center of the room, unobstructed.
    • Keep 2x2x2 studs of clear volume around it; no stuff within that area, think of this as like the nodes personal space.

    Waypoints
    • Create the top-level folder: Waypoints.
    • Only required if Start and Exit are not line-of-sight.
    • Inside Waypoints, add one folder per exit: 1, 2, …
    • Drop waypoint parts in order inside each subfolder. Keep them named naturally or numbered.
    Room Model
      └─ Waypoints
        ├─ 1
        │   ├─ WP1
        │   └─ WP2
        └─ 2
            └─ WP2

    Folders

    #

    Required tree

     
    Room Model
      ├─ Assets          - structure: Walls, Floor, Ceiling, Windows/Frames
      ├─ Furniture       - anything with a Part named "Primary" inside the model
      ├─ Lights          - all light sources (bulbs, sconces, strips)
      ├─ RoomHitbox      - convex/union set that fully contains the room
      ├─ Scriptables     - gates, triggers, interactions
      ├─ Waypoints       - path folders (1, 2, ...)
      ├─ RoomModule      - ModuleScript: enter/exit hooks (remove if unused)
      ├─ Start           - entry locator (faces in)
      ├─ Exit            - exit locator (faces out)
      └─ RoomEntity_SpawnNode
    • Do not rename these core folders/objects.
    • Structure inside Assets can be further sub-organized, but keep it sane.

    Furniture rule
    • Any model containing a Part named Primary is classified as Furniture and must live in Furniture.
    • Use the Multitool to convert raw models into Furniture models.

    RoomHitbox

    #

    Coverage
    • Hitbox must fully contain the entire playable room including trims that block players.
    • No holes, no leaks, no narrow slivers. Prefer clean boxes/unions over spaghetti.
    • Keep hitbox parts non-collidable and in the RoomHitbox folder only.

    Lighting

    #

    Placement & Spacing
    • Lightbulbs live in Lights folder only. Do not change the PrimaryPart of any light prefab.
    • Baseline spacing: ~20 studs between bulbs; adjust for even coverage and shadows.
    • Vertical: bulbs are 3.5 studs below ceiling (bottom of bulb → top of wire). Use judgement on tall ceilings.

    Performance
    • Keep shadow-casters to the minimum necessary. Disable shadows on tiny fixtures.
    • Keep per-room light count within budget (document your max once finalized).

    Pillars

    #

    Spacing
    • Default pillar pitch: 10 studs (center to center) unless the room’s module specifies otherwise.
    • All pillar bases align to floor grid; caps align with ceiling modules.

    Positions

    #

    Snap rules
    • Never scale/move/rotate structure freely. Always snap to grid.
    • Decor exceptions: minor rotation/offset only; keep on-grid for bounding/collision.

    Light prefab edits
    • Do not change PrimaryPart of any light prefab.
    • If internal parts are edited (mesh/size), notify Fancy Ducc or Lighty.

    RoomModule

    #

    When to include it
    • Only if the room needs logic (gates, events, VFX).
    • If unused, remove the ModuleScript.

    Template code

     
    local RoomModule = {}

local Room : Instance = script.Parent

function RoomModule.Start()
    -- Code Here
end

return RoomModule

    Checklist

    #
    • ✅ All transforms are multiples of 0.125
    • ✅ no z-fighting AT ALL (even ones too small to see unless you're looking for it)
    • ✅ Start faces outside of the room.
    • ✅ Exit faces inside of the room.
    • ✅ SpawnNode present and clear.
    • ✅ Waypoints added only when Start↔Exit are not in clear view; foldered by exit number.
    • ✅ Folder tree matches spec; lights in Lights, furniture in Furniture.
    • ✅ RoomHitbox fully contains room with no leaks, the only parts that should be outside of the hitbox are the Start and Exit parts
    • ✅ Lighting spacing roughly 20 studs and light bulbs 3.5 studs below ceiling (or justified).

    Common Mistakes

    #
    • ❌ Lights/furniture mixed into Assets.
    • ❌ Start/Exit backwards, or floating off the threshold.
    • ❌ Hitbox too small.
    • ❌ Off-grid trims causing hairline seams and z-fighting.
